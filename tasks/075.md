## Context

The current `DevContainerRuntime` reuses containers between agent runs. Multiple conversations can hit the same container simultaneously, racing over the same filesystem. There is no project identity, no git history, and no way to roll back.

This task refactors the runtime to be **session-based and ephemeral**:

1. Session starts → spin up a fresh container
2. Container clones the project's git repo into `/workspace`
3. Agent does its work (code, build, test locally, deploy InsForge functions)
4. Agent calls `serve_workspace_app()` to promote the build (updates bind mount → app goes live)
5. Session ends → `git add -A && git commit && git push` → container is destroyed

The **workspace bind mount** (not the container) is the persistent layer. It holds the last promoted static build and serves the live app via the app proxy between sessions.

## Sessions Data Model

### Migration: `022_project_sessions.sql`

```sql
CREATE TABLE project_sessions (
```

## Runtime Refactor

### New: `ProjectSessionRuntime` (`src/runtime/project-session.ts`)

Replaces `DevContainerRuntime` for project-based execution.

```
execute(projectId, agentId, userId, input):
  1. Create session record (status: pending)
  2. Load project + workspace from DB
  3. Start fresh container (no reuse)
  4. Clone project repo into /workspace
  5. Run AgentService.processStream()
  6. Stream events to caller
  7. On completion:
     a. git add -A && git commit -m "{commit_message}"
     b. git push
     c. Record commit_sha in session
     d. Destroy container
     e. Session status → 'committed'
  8. On failure:
     a. Attempt git push anyway (partial work is better than nothing)
     b. Session status → 'failed'
     c. Destroy container
```

### Container startup

```typescript
async startSessionContainer(project: ProjectRow, workspace: WorkspaceRow): Promise<string> {
  // 1. Create container (same as current WorkspaceManager.start())
  //    - image: DEVCONTAINER_IMAGE
  //    - mount: {workspace.volume_name}:/workspace (bind mount)
  //    - memory limit, cpu shares from env
  //    - extra_hosts: host.docker.internal:host-gateway
  //    - label: tela.project-id, tela.session-id

  // 2. Clone the project repo
  //    docker exec {containerId} git clone {GIT_SERVER_CLONE_URL}/{slug}.git /workspace/repo
  //    (Note: /workspace/repo is the source; /workspace/dist or similar is the build output)

  // 3. Configure git identity inside container
  //    docker exec {containerId} git -C /workspace/repo config user.name "Tela App Builder"
  //    docker exec {containerId} git -C /workspace/repo config user.email "builder@tela.internal"

  return containerId;
}
```

### Working directory convention

```
/workspace/
  repo/          ← git working tree (cloned from Gitea)
    src/         ← source code
    dist/        ← built frontend (after npm run build)
    ...
  serve/         ← symlink or copy of dist/ after serve_workspace_app() is called
                 ← this is what static_app_path points to
```

The separation between `repo/` and `serve/` means:
- The app proxy always serves from `serve/` (stable, last promoted build)
- The agent works in `repo/` (in-progress, possibly broken mid-session)
- `serve_workspace_app("repo/dist")` copies/symlinks → `serve/` to promote

### Session commit logic

```typescript
async commitAndPush(containerId: string, repoPath: string, message: string): Promise<string> {
  // docker exec git -C {repoPath} add -A
  // docker exec git -C {repoPath} commit -m "{message}" --allow-empty-message
  // docker exec git -C {repoPath} push origin main
  // docker exec git -C {repoPath} rev-parse HEAD → returns commit SHA
}
```

Commit message is auto-generated from the agent's output summary (first 72 chars of final response).

### Container destruction

Always destroy after session — no pause/resume for session containers:

```typescript
async destroySessionContainer(containerId: string): Promise<void> {
  await docker.getContainer(containerId).stop({ t: 5 });  // 5s grace
  await docker.getContainer(containerId).remove();
}
```

The bind mount (workspace directory on host) is NOT deleted — it holds the live static build.

## serve_workspace_app changes

Currently: sets `static_app_path` directly in DB to a path inside the bind mount.

After this change: the tool copies the build output to the stable `serve/` directory in the bind mount, then updates `static_app_path`:

```typescript
// Inside the container, the tool calls back to Tela API:
POST /api/workspaces/{workspaceId}/serve
  { source_directory: "repo/dist", api_port?: number }

// Tela API:
// 1. Copy {bindMount}/repo/dist → {bindMount}/serve (rsync or recursive copy)
// 2. Set workspace.static_app_path = 'serve'
// 3. If api_port: allocate port proxy
```

This is the zero-downtime guarantee: old `serve/` stays live until the copy completes.

```
  {git log --oneline -5}

Working directory: /workspace/repo
Build output: /workspace/repo/dist (or wherever vite outputs)

IMPORTANT:
- All your changes live in /workspace/repo
- git is already configured — do NOT change git user/email
- At the end of your session, call serve_workspace_app("repo/dist") to deploy
- Do NOT call git commit or git push yourself — the runtime does this automatically
```

The `POST /api/projects/:id/sessions` route is the new entry point for "chat with this project's agent."

## Orphan cleanup on startup

On Tela server startup, scan Docker for containers with label `tela.project-session` that are still running — these are orphans from a crash. For each:

1. Attempt git push (save partial work)
2. Destroy the container
3. Mark session as `failed` with error `'server_restart'`

## Acceptance Criteria

- [ ] Starting a session creates a fresh container (never reuses)
- [ ] Container has the project repo cloned at `/workspace/repo` on startup
- [ ] Agent can read/write files, run `npm install`, `npm run build` inside container
- [ ] `serve_workspace_app("repo/dist")` makes the build live immediately via app proxy
- [ ] On session success: git commit + push happens, commit SHA recorded in session
- [ ] On session failure: git push attempted, session marked `failed`, container destroyed
- [ ] Container is ALWAYS destroyed after session (success or failure)
- [ ] Workspace bind mount persists after container destruction (static files stay live)
- [ ] Two sessions for the same project cannot run simultaneously (second is queued or rejected)
- [ ] Orphan containers from previous crash are cleaned up on server start
- [ ] `GET /api/projects/:id/sessions` returns session history with commit SHAs
- [ ] Project context block appears in agent's system prompt

## Non-Goals

- Session queuing (for now: reject concurrent sessions with 409)
- Branch-per-session (always pushes to main)
- PR review flow before deploy (future task)
- Container resource metrics per session
