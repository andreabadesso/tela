# 064 — Deferred Tool Loading

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 042
**Estimated effort**: medium (2-8h)

## Context

Currently all MCP tools (vault tools + knowledge source tools + external MCP tools) are loaded upfront and sent to the model in the system prompt. As we add more knowledge sources and MCP connections, this bloats the prompt with tool schemas the model may never use. The cc-tips docs describe deferred loading: send only tool *names* initially, and the model calls a `ToolSearch` meta-tool to load full schemas on demand.

**Source:** cc-tips `TOOL_PIPELINE.md` (assembleToolPool, deferred loading section)

## Deliverables

### 1. Tool registry with deferred schemas

Split tool definitions into two tiers:

**Always loaded (small set, always relevant):**
- Core vault tools: `read_note`, `write_note`, `search_vault`, `list_notes`
- The `tool_search` meta-tool itself

**Deferred (loaded on demand):**
- Knowledge source tools: `search_<source>`, `read_<source>`, `list_<source>` per source
- External MCP tools: all tools from external servers (ShipLens, Jira, GitHub, etc.)
- Specialized vault tools: `edit_note`, `append_to_note`, `prepend_to_note`, `get_tasks`, `get_daily_note`

### 2. ToolSearch meta-tool

A built-in tool the model can call to discover and load tool schemas:

```typescript
tool("tool_search", {
  description: "Search for available tools by keyword. Returns full tool schemas so you can call them.",
  parameters: z.object({
    query: z.string().describe("Keywords to search for tools"),
    max_results: z.number().default(5),
  }),
  execute: async ({ query, max_results }) => {
    // Score each deferred tool against query
    // Return top N with full schemas
    // Scoring: exact name match (10) > keyword in name (5) > keyword in description (2)
  }
});
```

### 3. Tool catalog in system prompt

Instead of full schemas for deferred tools, include a compact catalog:

```
Available tools (call tool_search to load):
- search_engineering_docs: Search engineering documentation
- search_product_docs: Search product documentation
- shiplens_developer_scores: Get developer productivity scores
- jira_search_issues: Search Jira issues
- github_list_prs: List pull requests
... (one line per tool)
```

This gives the model enough context to know what's available without the full schema cost.

### 4. Schema caching

Once a tool schema is loaded via `tool_search`, cache it for the rest of the session:
- Don't require re-loading on subsequent calls
- Include loaded schemas in compaction restoration (task 061)

### 5. Token savings tracking

Measure prompt size before/after deferred loading:
- Log: tools_loaded_upfront, tools_deferred, tokens_saved_per_turn
- Expected: 40-60% reduction in tool schema tokens for users with 3+ knowledge sources

## Architecture Decision

**Why not just always load everything?**

With 3 knowledge sources × 3 tools each + 8 vault tools + N external MCP tools, we're looking at 20-30 tool schemas in every prompt. Each schema is ~200-500 tokens. That's 4K-15K tokens per turn just for tools the model may never use. Deferred loading brings this down to ~1K (catalog + core tools).

## Acceptance Criteria

- [ ] Core vault tools always loaded
- [ ] Knowledge + external MCP tools deferred
- [ ] `tool_search` meta-tool works: model can discover and call deferred tools
- [ ] Compact catalog in system prompt (one line per deferred tool)
- [ ] Schema caching: loaded tools stay available for session
- [ ] Token savings logged and measurable (>30% reduction)
- [ ] No UX regression: model still finds and uses the right tools
