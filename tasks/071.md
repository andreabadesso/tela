# 071 — Agent Memory System

**Status**: done
**Phase**: 9 — Agent Intelligence & Personalization
**Dependencies**: 042 (multi-agent), 070 (runtime abstraction)
**Effort**: large

## Summary

Persistent memory system for agents — automatic extraction, typed memories, relevance-based retrieval, user preferences, and behavior configuration. Inspired by Claude Code's auto-memory architecture, adapted for Tela's multi-agent, multi-user platform.

## Architecture Overview

```
Agent processes a message
  └─ Context injection: load memory index + relevant memories into system prompt
       └─ Agent responds (with memory MCP tools available)

Post-response (fire-and-forget)
  └─ Auto-extraction: cheap model call extracts new memories from conversation
       └─ Write to agent_memories table + update memory index
```

## Data Model

### Migration: `011_agent_memory.sql`

```sql
-- Core memory storage
CREATE TABLE agent_memories (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL REFERENCES agents(id),
  user_id TEXT REFERENCES users(id),        -- NULL = global agent memory
  scope TEXT NOT NULL DEFAULT 'global',      -- 'global' | 'user'
  type TEXT NOT NULL,                        -- 'user' | 'feedback' | 'project' | 'reference' | 'preference'
  name TEXT NOT NULL,                        -- short title
  description TEXT NOT NULL,                 -- one-line summary (used for relevance matching)
  content TEXT NOT NULL,                     -- full memory content (markdown)
  source TEXT DEFAULT 'auto',               -- 'auto' | 'manual' | 'tool'
  stale_after_days INTEGER,                 -- optional: mark stale after N days
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now'))
);

CREATE INDEX idx_agent_memories_agent ON agent_memories(agent_id);
CREATE INDEX idx_agent_memories_user ON agent_memories(user_id);
CREATE INDEX idx_agent_memories_scope ON agent_memories(agent_id, scope);
CREATE INDEX idx_agent_memories_type ON agent_memories(type);

-- User-configurable agent behavior
CREATE TABLE agent_behavior_config (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL REFERENCES agents(id),
  user_id TEXT REFERENCES users(id),        -- NULL = default for all users
  config TEXT NOT NULL DEFAULT '{}',         -- JSON: tone, language, verbosity, etc.
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now')),
  UNIQUE(agent_id, user_id)
);
```

### Memory Types

| Type | Scope | When to Save | Example |
|------|-------|-------------|---------|
| `user` | per-user | Learn about user's role, preferences, expertise | "Andre is CTO, deep in blockchain" |
| `feedback` | global or per-user | User corrections AND confirmations | "Don't summarize at end of responses" |
| `project` | global | Goals, deadlines, incidents, decisions | "Merge freeze March 5 for mobile release" |
| `reference` | global | Pointers to external systems | "Bugs tracked in Linear project INGEST" |
| `preference` | per-user | How user wants the agent to behave | "Prefers concise answers, no emojis" |

## Memory MCP Tools

Expose to agents via new MCP server (like vault-tools):

### `remember`
```
remember(content: string, type: 'user'|'feedback'|'project'|'reference'|'preference', name: string, scope?: 'global'|'user')
```
Save a memory. If scope='user', requires userId from input context.

### `recall`  
```
recall(query: string, scope?: 'global'|'user'|'all', type?: string, limit?: number)
```
Search memories by text. Returns matching memories sorted by relevance.

### `forget`
```
forget(memoryId: string)
```
Delete a specific memory.

### `list_memories`
```
list_memories(scope?: 'global'|'user'|'all', type?: string)
```
List all memories, optionally filtered.

### `get_user_context`
```
get_user_context()
```
Returns: user preferences, behavior config, recent interaction summary.

## Context Injection (in AgentService.process)

At query time, inject into system prompt BEFORE the user's message:

```
1. Agent system prompt (existing)
2. Date, language preference (existing)
3. === MEMORY CONTEXT ===
   a. Global memories for this agent (all types, cap 50)
   b. User-specific memories (if userId present, cap 30)
   c. User behavior config (if exists)
   d. Staleness warnings for memories > stale_after_days
4. Recent conversation history (existing)
```

**Cap**: Total memory context injected should not exceed ~4000 tokens (~16KB). Truncate oldest/lowest-priority memories first.

### Relevance Filtering (Phase 2 — optional for v1)

For v1: inject ALL memories (with caps). 
For v2: use a cheap side-query to rank memories by relevance to current input, inject top N.

## Auto-Extraction (Post-Response)

After `AgentService.process()` completes, fire-and-forget:

1. **Gate checks**: 
   - Memory feature enabled (env: `AGENT_MEMORY_ENABLED=true`, default true)
   - Not a cron/background source (only 'web', 'telegram', 'agent')
   - Throttle: extract every N turns (default: 1 — every turn)

2. **Extraction call**:
   - Use a cheap/fast model (haiku or sonnet) 
   - Send: last user message + agent response + existing memory manifest
   - Prompt: "Extract any new memories from this conversation. Types: user, feedback, project, reference, preference. Do NOT extract: code patterns, git history, debugging solutions, ephemeral details."
   - Response format: JSON array of `{ type, name, description, content, scope }`

3. **Deduplication**:
   - Before saving, check existing memories for similar name+description
   - If duplicate found, UPDATE instead of INSERT (merge content)

4. **What NOT to extract** (enforced in prompt):
   - Code patterns, conventions, architecture
   - Git history, who-changed-what
   - Debugging solutions (the fix is in the code)
   - Ephemeral task details, temporary state
   - Things already in the agent's system prompt

## Acceptance Criteria

1. **Memory MCP tools work**: Agent can `remember`, `recall`, `forget`, `list_memories`
2. **Auto-extraction**: After each web/telegram conversation, memories are extracted automatically
3. **Context injection**: Memories are injected into system prompt at query time
4. **Per-user scoping**: User-specific memories isolated, global memories shared
5. **Behavior config**: Users can set tone, language, verbosity preferences per agent
6. **API routes**: Full CRUD for memories and preferences
7. **Staleness**: Old memories get freshness warnings
8. **Deduplication**: No duplicate memories created
9. **What NOT to save**: Extraction prompt enforces exclusions
10. **Non-blocking**: Extraction runs fire-and-forget, never blocks responses

## Non-Goals (for this task)

- Relevance-based retrieval via side-query (Phase 2)
- Team/shared memory sync
- Nightly consolidation/auto-dream
- Memory promotion workflow
- Vector-based semantic search on memories (use text LIKE for v1)
