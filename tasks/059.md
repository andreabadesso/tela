# 059 — Obsidian Knowledge Base Integration (Governed)

**Status**: pending
**Phase**: 7 — Knowledge
**Dependencies**: 041, 050, 052
**Estimated effort**: large (8-20h)

## Context

The Obsidian vault is currently hardcoded via `VAULT_PATH` env var and injected into every agent with full read/write access. This task makes Obsidian a proper governed knowledge source — configurable from the UI, with folder-level access control, proper frontmatter parsing, heading-aware vector search, and multi-vault support.

**Research conclusion:** Don't depend on Obsidian desktop running. Enhance the direct filesystem adapter with the features that matter: frontmatter, tags, wikilinks, heading-aware chunking, and folder-scoped access.

```typescript
import matter from 'gray-matter';

function parseNote(content: string) {
  const { data: frontmatter, content: body } = matter(content);
  return {
    frontmatter, // { tags: [...], aliases: [...], created: ..., custom: ... }
    body,
    tags: extractTags(frontmatter, body), // from frontmatter + inline #tags
    wikilinks: extractWikilinks(body),    // [[Note Name]] references
  };
}
```

#### Tag extraction (frontmatter + inline)

```typescript
function extractTags(frontmatter: Record<string, unknown>, body: string): string[] {
  const fmTags = Array.isArray(frontmatter.tags) ? frontmatter.tags : [];
  const inlineTags = [...body.matchAll(/#([a-zA-Z][\w/-]*)/g)].map(m => m[1]);
  return [...new Set([...fmTags, ...inlineTags])];
}
```

#### Wikilink extraction

```typescript
function extractWikilinks(body: string): string[] {
  return [...body.matchAll(/\[\[([^\]|]+)(?:\|[^\]]+)?\]\]/g)].map(m => m[1]);
}
```

#### Heading-aware chunking for vector search

Replace the current dumb 500-word chunking with:

1. Split on `## ` heading boundaries
2. Each chunk carries: note title, heading path (e.g., "Project Alpha > Architecture > Database"), tags, folder
3. Chunks > 2000 chars split on paragraph boundaries (double newline)
4. 200-char overlap between chunks

```typescript
interface VaultChunk {
  content: string;
  metadata: {
    file: string;
    folder: string;          // top-level folder for access control
    heading: string;         // full heading path
    tags: string[];
    lastModified: Date;
  };
}
```

#### Folder-scoped access

Each ObsidianAdapter instance can be configured with path scope:

```typescript
new ObsidianAdapter("engineering-docs", "/vaults/company", {
  rootScope: "/Engineering",     // only sees /Engineering/**
  allowedPaths: ["/Engineering", "/Shared"],
  deniedPaths: ["/Engineering/Confidential"],
  permissions: ['read', 'write', 'search'],
});
```

Every `read()`, `write()`, `list()`, `search()` call checks against the path policy before executing.

## Architecture Decision

**Why not use an existing Obsidian MCP server?**

| Existing Server | Why Not |
|----------------|---------|
| mcp-obsidian (Python) | Wrong language, requires Obsidian running |
| cyanheads/obsidian-mcp-server | Requires Obsidian Local REST API plugin running |
| StevenStavrakis/obsidian-mcp | Self-described as "not thoroughly tested" |
| mcpvault (bitbonsai) | No access control, no semantic search |

We borrow patterns from mcpvault (gray-matter, BM25 reranking) and alexhholmes/mcp-obsidian (ChromaDB architecture) but build our own because:
1. We need folder-level access control (no existing server has this)
2. We need governance integration (MCP Gateway + RBAC policies)
3. We already have 80% of the implementation
4. We don't want to depend on Obsidian desktop running
