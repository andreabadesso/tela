# 073 — Local Git Server (bare repos over HTTP)

**Status**: pending
**Phase**: 8 — Agent Orchestration & Isolation
**Dependencies**: 046 (Docker production build)
**Estimated effort**: small

## Context

The App Builder agent needs a remote git server so that:
- Any agent with permission can clone a project, work, and push changes back
- Each session starts from a clean clone of HEAD — no shared state between sessions
- Project history is preserved across container restarts and server reboots

We don't need a web UI. We need the simplest possible HTTP git server that supports push and pull over `http://`. No auth (internal network only — access control lives in the Tela API).

## Design

### Approach

A Docker container running **nginx + git-http-backend + fcgiwrap**. This is git's own built-in HTTP protocol handler. No third-party dependencies.

- Repos stored as bare git repos at `/repos/{slug}.git`
- Push enabled per-repo via `http.receivepack = true` git config
- Accessible at `http://tela-git:3500/{slug}.git` from other containers
- Accessible at `http://host.docker.internal:3500/{slug}.git` from devcontainers on Mac/Windows

### Docker Service

`docker/git-server/Dockerfile`:
```dockerfile
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y nginx fcgiwrap git && rm -rf /var/lib/apt/lists/*
COPY nginx.conf /etc/nginx/nginx.conf
RUN mkdir -p /repos && chown -R www-data:www-data /repos
EXPOSE 3500
CMD ["sh", "-c", "fcgiwrap -s unix:/var/run/fcgiwrap.sock & sleep 0.5 && nginx -g 'daemon off;'"]
```

`docker/git-server/nginx.conf`:
```nginx
user www-data;
worker_processes 1;
events { worker_connections 256; }

http {
  server {
    listen 3500;

    location ~ ^/(.+\.git)(/.*)$ {
      root /repos;
      fastcgi_pass  unix:/var/run/fcgiwrap.sock;
      fastcgi_param SCRIPT_FILENAME     /usr/lib/git-core/git-http-backend;
      fastcgi_param GIT_HTTP_EXPORT_ALL "";
      fastcgi_param GIT_PROJECT_ROOT    /repos;
      fastcgi_param PATH_INFO           $uri;
      fastcgi_param QUERY_STRING        $query_string;
      fastcgi_param REQUEST_METHOD      $request_method;
      fastcgi_param CONTENT_TYPE        $content_type;
      fastcgi_param CONTENT_LENGTH      $content_length;
      fastcgi_param SERVER_PROTOCOL     $server_protocol;
    }
  }
}
```

### Git volumes

Repos persist on a named Docker volume `tela_git_repos` mounted at `/repos` in the container. This survives container restarts.

### GitService (`src/services/git-service.ts`)

Tela's API manages repos by exec-ing into the container via Docker SDK:

```typescript
class GitService {
  async initRepo(slug: string): Promise<string> {
    // docker exec tela-git git init --bare /repos/{slug}.git
    // docker exec tela-git git -C /repos/{slug}.git config http.receivepack true
    // docker exec tela-git git -C /repos/{slug}.git symbolic-ref HEAD refs/heads/main
    return `http://tela-git:3500/${slug}.git`;
  }

  async deleteRepo(slug: string): Promise<void> {
    // docker exec tela-git rm -rf /repos/{slug}.git
  }

  async listRepos(): Promise<string[]> {
    // docker exec tela-git ls /repos
  }

  // URL devcontainers use to clone (host.docker.internal resolves on Mac/Windows/Linux with extra-hosts)
  cloneUrl(slug: string): string {
    return `http://host.docker.internal:${GIT_SERVER_PORT}/${slug}.git`;
  }

  // URL other containers on the Docker network use
  internalUrl(slug: string): string {
    return `http://tela-git:3500/${slug}.git`;
  }
}
```

### docker-compose additions

```yaml
  tela-git:
    build: ./docker/git-server
    container_name: tela-git
    volumes:
      - tela_git_repos:/repos
    ports:
      - "${GIT_SERVER_PORT:-3500}:3500"
    restart: unless-stopped
    networks:
      - tela

volumes:
  tela_git_repos:
```

### Environment variables

```
GIT_SERVER_URL=http://tela-git:3500          # for service-to-service (from Tela API)
GIT_SERVER_PORT=3500                         # host port binding
GIT_SERVER_CLONE_URL=http://host.docker.internal:3500  # for devcontainers to clone
GIT_SERVER_CONTAINER_NAME=tela-git           # for Docker exec calls
```

### Initial commit on repo creation

When a project is created, Tela pushes an initial commit so the repo is non-empty and cloneable:

```typescript
// In GitService.initRepo() after bare init:
// 1. Clone into a temp dir inside the container
// 2. Write README.md: "# {project_name}\n\nCreated by Tela App Builder."
// 3. git add + commit + push
// 4. Remove temp dir
```

This ensures agents can `git clone` immediately without special empty-repo handling.

## Deliverables

1. `docker/git-server/Dockerfile`
2. `docker/git-server/nginx.conf`
3. `src/services/git-service.ts` — `GitService` class (initRepo, deleteRepo, cloneUrl, internalUrl)
4. `docker-compose.yml` — add `tela-git` service + `tela_git_repos` volume
5. `src/config/env.ts` — add `GIT_SERVER_URL`, `GIT_SERVER_PORT`, `GIT_SERVER_CLONE_URL`, `GIT_SERVER_CONTAINER_NAME`
6. `src/index.ts` — instantiate `GitService`, pass to dependent services

## Acceptance Criteria

- [ ] `docker-compose up` starts `tela-git` container on port 3500
- [ ] `git clone http://localhost:3500/test.git` works after `GitService.initRepo('test')` is called
- [ ] `git push` to a cloned repo succeeds and changes appear in a fresh clone
- [ ] Repo data survives `docker-compose restart tela-git`
- [ ] `GitService.deleteRepo('test')` removes the repo and subsequent clone returns 404
- [ ] DevContainer can clone via `http://host.docker.internal:3500/{slug}.git`

## Non-Goals

- Authentication (internal network, access control is Tela's job)
- Web UI or repo browsing
- Webhooks / CI hooks (future)
- Git LFS support
