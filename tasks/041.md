# 041 — Knowledge Source Adapters

**Status**: pending
**Phase**: 5 — Platform
**Dependencies**: 037, 038, 029
**Estimated effort**: large (8-20h)

## Context

Knowledge is hardcoded to a local Obsidian vault. This task makes the knowledge layer pluggable — connect any source, normalize into a common interface, vector-search across all of them.

## Deliverables

### KnowledgeAdapter interface

```typescript
interface KnowledgeAdapter {
  id: string;
  type: 'obsidian' | 'notion' | 'confluence' | 'filesystem' | 's3';

  // Read
  search(query: string, opts?: { maxResults?: number }): Promise<KnowledgeDocument[]>;
  read(path: string): Promise<KnowledgeDocument>;
  list(directory?: string, opts?: { recursive?: boolean }): Promise<string[]>;

  // Write (optional — not all sources support it)
  write?(path: string, content: string): Promise<void>;
  append?(path: string, content: string): Promise<void>;

  // Sync to vector store
  sync(): Promise<{ added: number; updated: number; deleted: number }>;
  getStatus(): AdapterStatus;
}

interface KnowledgeDocument {
  path: string;
  content: string;
  metadata: {
    source: string;       // adapter id
    lastModified: Date;
    author?: string;
    tags?: string[];
  };
}
```

### Knowledge Sources page (`web/src/pages/Knowledge.tsx`)

- List connected knowledge sources with: name, type, doc count, last sync, status
- "Add Source" → select type → configure (path for filesystem, API key for Notion, etc.)
- "Sync Now" button → triggers full re-index
- "Remove" button
- Show indexing progress during sync

### Unified search

- Agent tool `search_knowledge` queries ALL connected sources via vector store
- Agent tool `read_document` reads from the appropriate adapter based on source prefix
- Results include source attribution: "from Obsidian vault" or "from Confluence"

### MCP server for knowledge

Register knowledge tools as MCP server so agents access them naturally:

```
Tools:
  - search_knowledge(query, sources?, maxResults?)
  - read_document(source, path)
  - write_document(source, path, content)  — if adapter supports writes
  - list_documents(source, directory?)
```
