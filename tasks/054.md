# 054 — Per-User Audit Trail

**Status**: pending
**Phase**: 6 — Enterprise
**Dependencies**: 048, 050
**Estimated effort**: small (1-4h)

## Context

Every agent action must be tied to the user who initiated it. The audit log becomes the compliance backbone — "who accessed what, when, and was it authorized?"

## Deliverables

### Enhance audit logging

Every audit entry now includes:
- `user_id` — who initiated the action
- `connection_id` — which MCP server was accessed
- `tool_name` — which specific tool
- `access_decision` — 'allowed', 'denied', 'rate_limited'

### Update audit middleware

The `AuditLogger` in `src/api/middleware/audit.ts` receives user context from the auth middleware and passes it through to all log calls.

### User-scoped audit views

- **Admin**: sees all audit entries, can filter by user
- **Regular user**: sees only their own entries

Update `GET /api/audit` to:
1. If user is admin → return all (existing behavior + user filter)
2. If regular user → return only entries where `user_id = currentUser.id`

### Enhanced audit log page

Update `web/src/pages/AuditLog.tsx`:
- Add user filter dropdown (admin only)
- Show user name/avatar next to each entry
- Add access_decision badge (allowed=green, denied=red, rate_limited=yellow)
- Add connection and tool name columns

### Per-user cost tracking

The `cost_events` table already has `user_id` (added in 047). Ensure:
- Each `query()` call logs cost with `user_id`
- Budget policies can scope to users, not just agents
- Admin can see cost breakdown by user

## Acceptance Criteria

- [ ] Every audit entry has user_id
- [ ] Denied MCP access logged as audit entry with access_decision='denied'
- [ ] Regular users see only their own audit entries
- [ ] Admins can filter by user
- [ ] Cost events tracked per user
- [ ] Audit log UI shows user, tool, connection, decision columns
