# 072 — App Proxy: RBAC-Controlled Access to Agent-Generated Applications

**Status**: pending
**Phase**: 8 — Agent Orchestration & Isolation
**Dependencies**: 049 (RBAC), 050 (MCP governance), 070 (runtime abstraction)
**Estimated effort**: large

## Context

When a coding agent builds an app inside a DevContainer and calls `expose_port(3000)`, the current `PortProxyManager` creates a raw TCP proxy on `localhost:9000-9099` with **zero authentication**. Anyone on the network can access the app directly.

Delegating auth to the LLM ("build login into the app") is fragile — the agent will forget, misconfigure, or break it on the next iteration. **Auth for generated apps must be infrastructure, not application code.**

## Design

### Core Idea

Replace direct port access with an HTTP reverse proxy routed through the existing Hono server. All app traffic flows through `/apps/{workspaceId}/*`, which enforces Tela's session auth and workspace-level RBAC before forwarding.

```
Browser (Tela session cookie)
  │
  GET /apps/ws_abc/dashboard
  │
  ├─ Auth middleware (existing) → resolves AuthUser
  ├─ Workspace access check → owner_id / team_id / visibility
  ├─ Strip incoming X-Tela-* headers (anti-spoofing)
  ├─ Inject X-Tela-User-Id, X-Tela-User-Email, X-Tela-User-Roles
  ├─ Audit log (who accessed which app)
  │
  └─ HTTP proxy → localhost:{internalPort} → container app
```

### Workspace Access Model

New columns on `workspaces`:

| Column | Type | Default | Purpose |
|--------|------|---------|---------|
| `owner_id` | TEXT REFERENCES users(id) | NOT NULL | Who created this workspace |
| `visibility` | TEXT | 'private' | 'private' \| 'team' \| 'public' |
| `team_id` | TEXT REFERENCES teams(id) | NULL | For team visibility |
| `jwt_secret` | TEXT | NOT NULL | Per-workspace HS256 secret (encrypted) |

Access rules:
- `private` → only `owner_id`
- `team` → any member of `team_id`
- `public` → any authenticated Tela user
- Admins always have access
- Default is **private** (RBAC principle: default deny)

### URL Scheme

Single port per workspace (most common):
```
/apps/{workspaceId}/*
```

Multiple exposed ports:
```
/apps/{workspaceId}/_port/{containerPort}/*
```

The default port is the first entry in `port_mappings`. Explicit `_port/N` selects a specific container port.

### HTTP Proxy Route (`src/api/routes/app-proxy.ts`)

```typescript
export function appProxyRoutes(deps: { db: DatabaseService; runtimeRegistry: RuntimeRegistry }) {
  const app = new Hono();

  app.all('/apps/:workspaceId/*', async (c) => {
    const user = c.get('user') as AuthUser;
    const workspace = deps.db.getWorkspace(c.req.param('workspaceId'));

    // 1. Workspace exists and is running?
    if (!workspace || workspace.status !== 'running') return c.json({ error: 'not found' }, 404);

    // 2. Access check
    if (!canAccessWorkspace(user, workspace, deps.db)) return c.json({ error: 'forbidden' }, 403);

    // 3. Resolve target port
    const mappings = JSON.parse(workspace.port_mappings);
    const target = resolveTargetPort(c.req.path, mappings);
    if (!target) return c.json({ error: 'no exposed port' }, 502);

    // 4. Intercept /__tela/* meta-endpoints
    const subpath = extractSubpath(c.req.path, workspace.id);
    if (subpath.startsWith('/__tela/')) {
      return handleTelaEndpoint(subpath, user, workspace, deps.db);
    }

    // 5. Strip spoofed headers, inject identity
    const headers = stripTelaHeaders(c.req.raw.headers);
    injectTelaHeaders(headers, user);

    // 6. Proxy to internal port
    const targetUrl = `http://127.0.0.1:${target.hostPort}${subpath}`;
    return proxyRequest(c.req.raw, targetUrl, headers);
  });

  return app;
}
```

### Access Check Function

```typescript
function canAccessWorkspace(user: AuthUser, workspace: WorkspaceRow, db: DatabaseService): boolean {
  // Admins always access
  if (user.roles.includes('admin')) return true;

  // Owner always accesses their own
  if (workspace.owner_id === user.id) return true;

  switch (workspace.visibility) {
    case 'public':
      return true; // any authenticated user
    case 'team':
      return workspace.team_id ? user.teams.includes(workspace.team_id) : false;
    case 'private':
    default:
      return false;
  }
}
```

### Reserved `/__tela/*` Endpoints

Intercepted by the proxy before reaching the app:

| Endpoint | Returns | Purpose |
|----------|---------|---------|
| `GET /__tela/me` | `{ id, email, name, roles, teams }` | Frontend identity |
| `GET /__tela/token` | `{ token, expires_at }` | Short-lived JWT for InsForge RLS |
| `GET /__tela/logout` | Redirect to `/` | Session end |

### JWT Minting for InsForge

Using `jose` (zero-dependency, Web Crypto):

```typescript
import { SignJWT } from 'jose';

async function mintTelaJwt(user: AuthUser, workspace: WorkspaceRow): Promise<string> {
  const secret = decryptJwtSecret(workspace.jwt_secret);
  return new SignJWT({
    sub: user.id,
    email: user.email,
    roles: user.roles,
    workspace_id: workspace.id,
    aud: 'authenticated',     // Supabase convention
    role: 'authenticated',    // Supabase convention
  })
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setIssuer('tela')
    .setExpirationTime('5m')
    .sign(secret);
}
```

The same secret is configured as the InsForge project's JWT secret, so Supabase RLS `auth.uid()` returns the Tela user ID.

### Identity Headers

Injected on every proxied request (stripped first to prevent spoofing):

```
X-Tela-User-Id: usr_123
X-Tela-User-Email: andre@company.com
X-Tela-User-Name: André
X-Tela-User-Roles: admin,engineering
X-Tela-User-Teams: platform
```

Backend apps read these directly. No SDK needed.

### Port Proxy Changes

The TCP proxy on 9000-9099 stays as an internal hop but gets locked down:
- Bind to `127.0.0.1` only (not `0.0.0.0`)
- External access goes through `/apps/*` only

### `expose_port` Tool Update

Returns the proxy URL instead of raw port:

```typescript
// Before:
return `Port ${port} exposed → http://localhost:${hostPort}`;

// After:
const baseUrl = process.env.BASE_URL || 'http://localhost:3000';
return `Port ${port} exposed → ${baseUrl}/apps/${workspaceId}`;
```

### Agent Instructions

The coding agent's system prompt gets a standard block:

> When building apps in Tela:
> - **Never implement authentication.** Tela handles it.
> - For the current user: `fetch('/__tela/me')`
> - For a Supabase auth token: `fetch('/__tela/token')` → use as Bearer token
> - Always add `user_id` column to tables with per-user data
> - Always enable RLS with `auth.uid()` policies

## Acceptance Criteria

- [ ] Unauthenticated requests to `/apps/*` return 401
- [ ] Authenticated user can access their own workspace's app
- [ ] Authenticated user CANNOT access another user's private workspace
- [ ] Team visibility allows team members, blocks non-members
- [ ] Public visibility allows any authenticated user
- [ ] Admins can access all workspaces regardless of visibility
- [ ] `GET /__tela/me` returns current user identity JSON
- [ ] `GET /__tela/token` returns valid JWT that InsForge/Supabase accepts for RLS
- [ ] `X-Tela-User-*` headers are injected on proxied requests
- [ ] Incoming `X-Tela-*` headers are stripped (anti-spoofing)
- [ ] WebSocket connections (HMR) work through the proxy
- [ ] `expose_port` tool returns `/apps/{workspaceId}` URL
- [ ] TCP proxy ports (9000-9099) are bound to 127.0.0.1 only
- [ ] Workspace list API filters by user access (users only see workspaces they can access)
- [ ] Default visibility is `private`
- [ ] Workspace owner is set automatically from the requesting user's context

## Non-Goals (Future)

- Signed share links (time-limited public URLs for demos)
- Per-app rate limiting
- Custom domains per workspace
- InsForge deployment proxy (InsForge deployments live on their own infra)
