# 070 — Agent Runtime Abstraction (Pluggable Execution Backends)

**Status**: done
**Phase**: 8 — Agent Orchestration & Isolation
**Dependencies**: 042, 046
**Estimated effort**: large (2-4 days)

## Context

Agents currently run **in-process** inside the Node.js server via `AgentService.process()`. This means no isolation between agents, no resource limits, no horizontal scaling, and a rogue tool call can crash the entire platform. We need an `AgentRuntime` interface that decouples *where* an agent runs from *how* it's orchestrated.

## Design

### Core Interface

```typescript
interface AgentRuntime {
  execute(params: AgentExecutionParams): Promise<AgentExecutionHandle>;
  status(runId: string): Promise<AgentRunStatus>;
  cancel(runId: string): Promise<void>;
  logs(runId: string): AsyncIterable<string>;
}

interface AgentExecutionParams {
  agentId: string;
  input: AgentInput;
  config: AgentRow;
  mcpServers: McpServerConfig[];
  timeout?: number;       // wall-clock ms, default 5 min
  resources?: ResourceLimits;
}

interface ResourceLimits {
  maxMemoryMb?: number;   // default 512
  maxCpuShares?: number;  // default 1024 (1 core)
  maxDiskMb?: number;     // default 256
}

interface AgentExecutionHandle {
  runId: string;
  stream: AsyncIterable<AgentStreamEvent>;  // for real-time output
  result: Promise<AgentOutput>;             // resolves when done
}

type AgentRunStatus =
  | { state: 'running'; startedAt: Date; turns: number }
  | { state: 'completed'; result: AgentOutput; duration: number }
  | { state: 'failed'; error: string; duration: number }
  | { state: 'cancelled'; duration: number }
  | { state: 'timeout'; partialResult?: AgentOutput; duration: number };

interface AgentStreamEvent {
  type: 'text' | 'tool_call' | 'tool_result' | 'error' | 'status';
  data: unknown;
  timestamp: number;
}
```

### Runtime Implementations

#### 1. `InProcessRuntime` (default, dev)

Current behavior wrapped in the interface. Runs `AgentService.process()` directly with a `setTimeout` for wall-clock timeout.

- **Pros**: Zero overhead, easy debugging, streaming works natively
- **Cons**: No isolation, shared memory, crash propagation
- **When**: Local dev, single-user instances, low-risk agents

#### 2. `DockerRuntime` (local production)

Spawns a container per agent run using the Docker Engine API (`dockerode`).

- Container image: same as main app, but entrypoint is `agent-worker` script
- Communication: mounted Unix socket or HTTP callback to host
- MCP access: host-networked proxy — agent container calls `host.docker.internal:{port}/mcp/{serverId}` which the main server proxies to real MCP connections (preserves governance)
- Vault access: bind-mount vault directory read-only; writes go through MCP vault-tools
- DB access: agent doesn't touch SQLite directly — reports results via callback
- Lifecycle: container created → started → monitored → result collected → removed

```typescript
class DockerRuntime implements AgentRuntime {
  async execute(params: AgentExecutionParams): Promise<AgentExecutionHandle> {
    const container = await this.docker.createContainer({
      Image: this.agentImage,
      Cmd: ['node', 'dist/agent-worker.js'],
      Env: this.buildEnv(params),
      HostConfig: {
        Memory: (params.resources?.maxMemoryMb ?? 512) * 1024 * 1024,
        CpuShares: params.resources?.maxCpuShares ?? 1024,
        NetworkMode: 'bridge',
        ExtraHosts: ['host.docker.internal:host-gateway'],
        AutoRemove: true,
      },
    });
    await container.start();
    // ... monitor, stream, collect result
  }
}
```

#### 3. `RemoteRuntime` (future, cloud)

Dispatches to a remote worker pool via HTTP/gRPC. Not implemented in this task — but the interface must accommodate it without breaking changes.

Possible backends: Kubernetes Jobs, ECS Tasks, Cloud Run Jobs, fly.io Machines.

### Agent Worker Process

New entrypoint `src/agent-worker.ts` that:

1. Reads execution params from env/stdin
2. Connects to host MCP proxy for tool access
3. Runs `query()` from claude-agent-sdk
4. Streams events back to host via callback URL
5. Posts final result and exits

```
┌─────────────────────┐         ┌──────────────────────────┐
│   Main Server        │         │   Agent Container         │
│                      │         │                           │
│  Orchestrator        │────────▶│  agent-worker.ts          │
│    │                 │  start  │    │                      │
│    │                 │         │    ├── query() claude SDK  │
│    │                 │◀────────│    ├── stream events       │
│    │  MCP Proxy ◀────│─ ─ ─ ──│────┤  MCP calls via proxy  │
│    │                 │         │    │                      │
│    │                 │◀────────│    └── POST /result        │
│  Store result        │  done   │                           │
└─────────────────────┘         └──────────────────────────┘
```

### MCP Proxy

The main server exposes an internal HTTP endpoint that the agent container uses to access MCP servers:

```
POST /internal/mcp-proxy/{serverId}/call
Authorization: Bearer {run-scoped-token}
Body: { method, params }
```

This preserves the governance layer — the proxy enforces the same RBAC/policy filtering as direct access. The agent container never connects to MCP servers directly.

### Runtime Selection

Configured via environment or per-agent:

```typescript
// Global default from env
AGENT_RUNTIME=in-process | docker | remote

// Per-agent override in agent config
{ "runtime": "docker", "resources": { "maxMemoryMb": 1024 } }
```

The orchestrator resolves runtime:
```typescript
function resolveRuntime(agent: AgentRow): AgentRuntime {
  const override = agent.permissions?.runtime;
  const global = process.env.AGENT_RUNTIME ?? 'in-process';
  const choice = override ?? global;
  return runtimes[choice];
}
```

### Run Tracking

New `agent_runs` table:

```sql
CREATE TABLE agent_runs (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL REFERENCES agents(id),
  runtime TEXT NOT NULL,          -- 'in-process' | 'docker' | 'remote'
  status TEXT NOT NULL DEFAULT 'pending',
  input TEXT NOT NULL,            -- JSON
  output TEXT,                    -- JSON, null until complete
  container_id TEXT,              -- Docker container ID if applicable
  started_at TEXT,
  completed_at TEXT,
  duration_ms INTEGER,
  error TEXT,
  resource_usage TEXT,            -- JSON: { peakMemoryMb, cpuSeconds }
  created_at TEXT DEFAULT (datetime('now'))
);
```

## Deliverables

### 1. `AgentRuntime` interface and types
Define in `src/types/runtime.ts`.

### 2. `InProcessRuntime`
Wrap current `AgentService.process()` in the runtime interface. Add wall-clock timeout via `AbortController`. This becomes the default — zero behavior change for existing users.

### 3. `DockerRuntime`
Implement container lifecycle: create, start, monitor, stream, collect, cleanup. Use `dockerode` for Docker Engine API. Handle container crashes, OOM kills, and timeouts gracefully.

### 4. Agent Worker entrypoint
`src/agent-worker.ts` — standalone process that executes a single agent run and reports back.

### 5. MCP Proxy endpoint
`/internal/mcp-proxy` route that forwards MCP calls from containers through the governance layer.

### 6. `agent_runs` table and migration
Track all executions regardless of runtime.

### 7. Orchestrator integration
Update `orchestrator.assign()` and `orchestrator.chat()` to use the runtime interface instead of calling `AgentService.process()` directly.

### 8. Runtime selection config
Env-based default + per-agent override in agent config.

## Acceptance Criteria

- [ ] `AgentRuntime` interface defined with `execute`, `status`, `cancel`, `logs`
- [ ] `InProcessRuntime` passes all existing tests with no behavior change
- [ ] `DockerRuntime` can execute an agent in a container with memory/CPU limits
- [ ] Agent container communicates with host MCP proxy for tool access
- [ ] Governance layer enforced for containerized agents (same RBAC as in-process)
- [ ] `agent_runs` table tracks all executions with status, duration, resource usage
- [ ] Wall-clock timeout kills agent runs that exceed the limit
- [ ] Container crashes and OOM kills are detected and reported cleanly
- [ ] Orchestrator uses runtime interface — no direct `AgentService.process()` calls
- [ ] Runtime selectable via `AGENT_RUNTIME` env var and per-agent config
- [ ] Docker runtime gracefully degrades to in-process if Docker is unavailable
