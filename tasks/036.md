# 036 — API Layer

**Status**: pending
**Phase**: 5 — Platform
**Dependencies**: 005
**Estimated effort**: medium (2-8h)

## Context

The agent currently only talks through Telegram. To support a web UI, we need an API layer that exposes conversations, agent management, and system status over HTTP + WebSocket.

## Deliverables

### HTTP API (Hono)

```
GET    /api/health                    — system status
POST   /api/chat                      — send message to agent, returns response
GET    /api/conversations             — list conversations (paginated)
GET    /api/conversations/:id         — single conversation with messages
DELETE /api/conversations/:id         — delete conversation
GET    /api/agents                    — list configured agents
POST   /api/agents                    — create agent
PUT    /api/agents/:id               — update agent config
DELETE /api/agents/:id               — delete agent
```

### WebSocket (streaming)

```
WS /api/chat/stream — real-time agent output
  → Send: { text, agentId }
  ← Receive: { type: 'thinking' | 'tool_call' | 'text' | 'done', data }
```

The agent's `process()` method needs to yield intermediate events (thinking, tool calls) not just final result.

### Auth

Simple token-based for now. Single `API_TOKEN` env var. Bearer header. Enough for self-hosted single-user.

### Refactor

- Extract agent processing into a service that both Telegram handler and API call
- Both interfaces use the same `agent.process()` underneath
- Response includes: text, toolCalls (list), durationMs, model used

## Acceptance Criteria

- [ ] Hono server running alongside Telegram bot
- [ ] All REST endpoints functional with JSON responses
- [ ] WebSocket streaming works — client sees thinking/tool_call/text events in real-time
- [ ] Auth via Bearer token
- [ ] Telegram and API share the same agent processing logic
- [ ] Conversations persisted to SQLite (already exists, extend schema if needed)
- [ ] Health endpoint returns: uptime, connected services, last agent activity
