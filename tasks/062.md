# 062 — Fork Cache Optimization for Multi-Agent

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 042, 060
**Estimated effort**: medium (2-8h)

## Context

When the orchestrator runs council mode (parallel agents on same query) or batch tasks, each agent builds its full context independently. This means N agents = N full prompt cache misses. The cc-tips docs describe a fork pattern where subagents share a byte-identical message prefix so the API prompt cache is hit once, and each agent only pays for its small directive.

**Source:** cc-tips `multi-agent/ARCHITECTURE.md`, `COST_TRACKING.md`

## Deliverables

### 1. Shared context prefix builder

When spawning multiple agents for the same query:

```typescript
function buildForkContext(query: string, sharedContext: Message[]): {
  sharedPrefix: Message[];  // Identical across all forks
  perAgent: Map<string, Message[]>;  // Agent-specific directive
} {
  // sharedPrefix: system prompt + conversation history + user query
  // perAgent: agent-specific system prompt + tool definitions
}
```

The key insight: the shared prefix must be **byte-identical** across all forks for the API cache to hit. This means:
- Same system prompt base
- Same conversation history
- Same user query
- Agent-specific instructions come AFTER the shared prefix

### 2. Update orchestrator council mode

Current council mode spawns agents independently. Change to:

1. Build shared prefix once
2. For each agent, append only the agent-specific directive
3. Send all requests with the shared prefix (API will cache after first request)
4. Cost: 1 full context + N small directives, instead of N full contexts

### 3. Update orchestrator batch mode

Same pattern for batch: shared context prefix + per-task directive.

### 4. Cost tracking for cache hits

Log cache hit/miss rates per session so we can measure the savings:
- `cache_creation_input_tokens` (25% of input cost)
- `cache_read_input_tokens` (10% of input cost)
- Track per-agent and per-session

## Architecture Decision

**Constraint:** The Claude API caches prompts based on exact prefix matching. Any byte difference in the prefix = cache miss. We must ensure the shared portion is deterministic (no timestamps, no random IDs in the prefix).

## Acceptance Criteria

- [ ] Council mode uses shared prefix — cache hit rate > 80% for 2nd+ agents
- [ ] Batch mode uses shared prefix
- [ ] Cost reduction measurable: N-agent council costs ~1.1x single agent, not Nx
- [ ] Cache hit/miss rates logged to cost_events table
- [ ] No behavior change: agents produce same quality results
