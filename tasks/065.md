# 065 — Circuit Breakers & Error Resilience

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 060
**Estimated effort**: small (1-4h)

## Context

The job registry already auto-disables after 3 failures, but no other subsystem has circuit breakers. The cc-tips docs describe a systematic approach: every subsystem explicitly classified as fail-open or fail-closed, with circuit breakers, retry strategies, and error watermarking.

**Source:** cc-tips `STREAMING_STATE_MACHINE.md`, development_practices docs

## Deliverables

### 1. Subsystem failure classification

Explicitly classify every subsystem:

| Subsystem | Failure Mode | Reason |
|-----------|-------------|--------|
| Claude API | retry + fallback | Core functionality |
| MCP external servers | fail-open | Agent works without external tools |
| Knowledge vector search | fail-open | Agent still has direct file access |
| ChromaDB | fail-open | Degrade to keyword search |
| WebSocket streaming | fail-closed | User must see responses |
| Auth/RBAC | fail-closed | Security critical |
| Audit logging | fail-open | Don't block agent on logging failure |
| Job scheduler | fail-open per job | One job failure doesn't stop others |
| Notification channels | fail-open | Don't block job on notification failure |

### 2. API retry with model fallback

When Claude API returns errors:

```typescript
const retryStrategy = {
  maxRetries: 5,
  backoff: 'exponential',  // 1s, 2s, 4s, 8s, 16s
  jitter: 0.25,            // ±25% randomization
  onOverload: (attempt: number) => {
    if (attempt >= 3) return fallbackModel(agent);
    // e.g., Opus → Sonnet after 3 failures
  },
};
```

### 3. Circuit breakers

Add circuit breakers to:

- **Auto-compact** (from task 061): 3 failures → stop trying, microcompact only
- **External MCP servers**: 3 consecutive failures → mark connection unhealthy, skip for 5 minutes, auto-retry
- **ChromaDB**: 3 failures → degrade to keyword search for session
- **Notification channels**: 3 failures → disable channel, alert admin

```typescript
class CircuitBreaker {
  private failures = 0;
  private lastFailure: Date | null = null;
  private state: 'closed' | 'open' | 'half-open' = 'closed';

  async execute<T>(fn: () => Promise<T>): Promise<T | null> {
    if (this.state === 'open') {
      if (this.shouldRetry()) this.state = 'half-open';
      else return null; // fail-open
    }
    try {
      const result = await fn();
      this.reset();
      return result;
    } catch (e) {
      this.recordFailure();
      throw e;
    }
  }
}
```

### 4. Error watermarking

Only surface errors newer than the last successful operation:

```typescript
class ErrorWatermark {
  private watermark: Date = new Date();

  markSuccess() { this.watermark = new Date(); }
  isRelevant(error: { timestamp: Date }) {
    return error.timestamp > this.watermark;
  }
}
```

This prevents stale errors from confusing the user (e.g., "ChromaDB error" from 10 minutes ago showing after it recovered).
