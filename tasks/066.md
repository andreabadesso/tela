# 066 — Structured Batch Workflows with Approval Gates

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 042, 060
**Estimated effort**: medium (2-8h)

## Context

The orchestrator supports batch mode with task checkout, but lacks approval gates and structured phases. The cc-tips docs describe a 3-phase batch pattern: (1) research/plan with human approval, (2) spawn workers, (3) track progress. The approval gate before spawning expensive work prevents wasted compute.

**Source:** cc-tips `BATCH_WORKFLOWS.md`

## Deliverables

### 1. Three-phase batch execution

Replace the current batch-assign-and-run with explicit phases:

**Phase 1 — Research & Plan:**
- Agent analyzes the batch request
- Produces a plan: what work units, which agent handles each, estimated cost
- Presents plan to user via WS message `{ type: "batch_plan", data: { units: [...] } }`
- Waits for approval: `{ type: "batch_approve" }` or `{ type: "batch_reject" }`

**Phase 2 — Spawn Workers:**
- Only after approval
- All workers spawned in a single orchestrator turn (for prompt cache sharing — task 062)
- Each worker gets a fully self-contained prompt (no "based on your findings" — everything the worker needs is in its prompt)

**Phase 3 — Track Progress:**
- Status table updated in real-time via WS:
  ```json
  {
    "type": "batch_status",
    "data": {
      "total": 5,
      "completed": 3,
      "in_progress": 1,
      "failed": 1,
      "units": [
        { "id": 1, "agent": "cto-agent", "status": "done", "result": "..." },
        { "id": 2, "agent": "cto-agent", "status": "running" },
        { "id": 3, "agent": "cto-agent", "status": "failed", "error": "..." }
      ]
    }
  }
  ```

### 2. Worker self-verification

Worker prompts include verification steps before reporting done:
- Review own output for quality
- Run any validation relevant to the task
- Report structured result (not free-form text)

### 3. Frontend batch UI

New component in chat view for batch operations:

- **Plan review card:** shows work units, estimated cost, approve/reject buttons
- **Progress table:** live-updating status per unit with expandable results
- **Summary card:** after completion, aggregated results

### 4. Batch cost estimation

Before spawning workers, estimate cost:
```typescript
function estimateBatchCost(units: WorkUnit[], model: string): number {
  const tokensPerUnit = 5000; // average input + output per work unit
  return units.length * tokensPerUnit * getModelCostPerToken(model);
}
```

Show this in the approval step so users know what they're about to spend.

### 5. Batch task persistence

Store batch state in DB so it survives server restarts:

```sql
CREATE TABLE batch_runs (
  id TEXT PRIMARY KEY,
  user_id TEXT,
  plan JSON,
  status TEXT, -- planning, approved, running, completed, failed, rejected
  units JSON,
  created_at TEXT,
  completed_at TEXT
);
```

## Acceptance Criteria

- [ ] Batch requests produce a plan requiring user approval
- [ ] Workers only spawn after explicit approval
- [ ] Worker prompts are fully self-contained
- [ ] Real-time progress updates via WebSocket
- [ ] Frontend shows plan review + progress table
- [ ] Cost estimation shown before approval
- [ ] Batch state persisted to DB (survives restarts)
- [ ] Failed units don't block other units
