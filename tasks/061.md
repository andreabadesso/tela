# 061 — Conversation Compaction

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 060
**Estimated effort**: large (8-20h)

## Context

Currently we pull the last 10 conversations as context — no intelligent compaction. As sessions get longer and agents use more tools, we'll hit context window limits or waste tokens on stale tool results. The cc-tips docs describe 6 compaction strategies layered by cost and urgency.

**Source:** cc-tips `COMPACTION.md`

## Deliverables

### 1. Microcompact (no LLM call)

Cheapest strategy — run every turn. Selectively clear old tool results from the conversation history:

- After N turns, replace tool result content with `[Result cleared — was: {summary}]`
- Targets: search results, file listings, long read outputs
- Keep: the tool *call* (so the model knows what it did) and a 1-line summary of the result
- Threshold: clear results older than 5 turns

```typescript
function microcompact(messages: Message[]): Message[] {
  // For each tool_result older than 5 turns:
  //   Replace content with truncated summary (first 200 chars + "...")
  //   Preserve tool_use_id for conversation coherence
}
```

### 2. Auto-compact (LLM call)

When context usage exceeds 70% of window:

1. Send conversation to a smaller model (Haiku) with a compaction system prompt
2. Generate a structured summary with sections:
   - Key decisions made
   - Current task state
   - Files/resources referenced
   - Pending actions
   - User preferences observed
3. Replace all messages before the summary with the summary message
4. Re-attach: recent files read, active plans, current tools

### 3. Reactive compact (emergency)

When the API returns `prompt_too_long`:

1. Strip the oldest 30% of messages
2. Retry the request
3. If still too long, fall back to auto-compact
4. If auto-compact also fails, start a fresh context with session summary

### 4. Circuit breaker

If auto-compact fails 3 consecutive times:
- Stop trying auto-compact for this session
- Fall back to microcompact only
- Log warning for observability

### 5. Post-compact restoration

After any compaction, automatically:
- Re-read the 3 most recently referenced files (so agent has fresh context)
- Re-attach active agent configuration
- Re-attach any in-progress tool state

### 6. Context usage tracking

Add context budget awareness to the streaming loop:
- Track token count per turn (input + output)
- Calculate % of context window used
- Trigger microcompact at 50%, auto-compact at 70%, reactive at API error

## Architecture Decision

**Compaction strategy priority:**
1. Microcompact — every turn, zero cost
2. Auto-compact — when needed, one Haiku call (~$0.001)
3. Reactive — emergency only, strips messages
4. Circuit breaker — prevents runaway costs
