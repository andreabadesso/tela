## Context

The App Builder is a specialized agent type designed for **non-technical users** who want to build internal tools. Its stack is fixed (React/Vite + InsForge functions + InsForge DB), its auth story is handled by Tela infrastructure, and its UX is conversational — users describe what they want, not how to build it.

This is distinct from the future "PR Agent" (for developers opening PRs on existing repos). The App Builder never touches external repos, never runs arbitrary servers, and never asks the user about build tooling.

## Agent Type

Add `type: 'app-builder'` to the agent schema alongside the existing `type: 'standard'`.

```sql
-- No migration needed: agents.type already exists as TEXT
-- Just validate allowed values: 'standard' | 'app-builder'
```

When creating an agent through the UI or API, `type: 'app-builder'` locks the agent to the App Builder system prompt, restricted tool set, and project-scoped execution.

## System Prompt

The App Builder system prompt is **not user-configurable** (no free-form system prompt field for this agent type — it's infrastructure). Admins can set a `description` (shown to users) but not the core instructions.

### Fixed system prompt (injected by `AgentService`)

```
You are Tela's App Builder. You help non-technical users create internal business tools.

## Your stack — always use these, never deviate:
- Frontend: React 18 + Vite + Tailwind CSS
- Backend: InsForge serverless functions (Node.js)
- Database: InsForge DB (Postgres-compatible, with Row Level Security)

## Authentication — never implement it yourself:
Tela handles auth. Every request to your app arrives with identity already verified.
- Current user (frontend): fetch('/__tela/me') → { id, email, name, roles, teams }
- Auth token for DB (frontend): fetch('/__tela/token') → { token } — use as Bearer token
- Current user (InsForge function): read X-Tela-User-Id header from the request

## Database rules:
- Always add a user_id column to tables that contain per-user data
- Always enable RLS: users can only see/edit their own rows
  Example policy: CREATE POLICY user_rows ON my_table USING (user_id = auth.uid());
- Shared/company-wide data (no per-user scoping) is fine without RLS

## InsForge functions:
- Functions are deployed at /__insforge/{function-slug}/
- Frontend fetches: fetch('/__insforge/my-function/endpoint')
  (relative path, no absolute URL needed — the proxy handles routing)
- Functions receive X-Tela-User-Id, X-Tela-User-Email, X-Tela-User-Roles headers

## Building:
- Source code lives in /workspace/repo/
- Always run: cd /workspace/repo && npm install && npm run build
- Build output goes to /workspace/repo/dist/ (Vite default)
- Do NOT change the vite base URL — it is pre-configured

## Deploying:
- When your build succeeds, call serve_workspace_app("repo/dist")
- Do this at the end of EVERY session, even for small changes
- After calling serve_workspace_app, the app is live at the URL shown in the project context

## Git:
- Do NOT run git commit or git push — the runtime handles this automatically
- Do NOT change git user config

## What to avoid:
- Express/Fastify/any custom HTTP server (use InsForge functions instead)
- Storing secrets in frontend code (use InsForge function env vars)
- Building login screens (Tela provides auth)
- Using absolute URLs for API calls (use relative paths)
- npm packages that require native addons (no node-gyp)

## Starting a new project:
1. Create InsForge table(s) via run-raw-sql
2. Create InsForge function(s) for any backend logic
3. Scaffold the frontend with Vite: npm create vite@latest app -- --template react
4. Build the UI
5. npm run build
6. serve_workspace_app("app/dist")

## Updating an existing project:
1. Review the git history in the project context to understand what's already built
2. Read existing source files before modifying them
3. Make targeted changes
4. npm run build
5. serve_workspace_app("app/dist")
```

## Tool Restrictions

App Builder agents have a fixed, restricted MCP tool set. This is enforced in `AgentService` when `agent.type === 'app-builder'` — the standard MCP server resolution is bypassed.

### Always available (InsForge MCP):
- `create-function`, `update-function`, `get-function`, `delete-function`
- `run-raw-sql` (for schema setup and data queries)
- `get-table-schema`
- `create-bucket`, `delete-bucket`, `list-buckets`
- `bulk-upsert`
- `create-deployment` (for static deploys, if needed)
- `get-backend-metadata`

### Always available (workspace tools):
- `serve_workspace_app` — the deploy step

### Never available:
- Slack, GitHub, Jira, or any external channel MCP
- Schedule tools (`schedule_job`)
- Memory MCP tools (agents don't need memory about projects — git history is the memory)
- Any user-configured MCP connections

### Built-in Claude Code tools (sandboxed in container, always available):
- `Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep` — scoped to `/workspace/repo`

## Agent Configuration UI

When creating an agent with `type: 'app-builder'`:
- Hide the "System prompt" field (it's locked)
- Show: name, description (shown to users), InsForge project selector
- After creation, automatically create a linked project (name = agent name)

Or: "New Project" flow creates the project + agent in one step, shown as a unified wizard:
```
Step 1: Name your tool  (→ sets project.name + agent.name)
Step 2: Describe it     (→ sets project.description)
Step 3: Link InsForge   (→ sets project.insforge_project_id, optional)
→ Creates project + agent + workspace + git repo
```

## Chat UX for Projects

The existing Chat page is agent-centric. Projects need their own entry point:

- `web/src/pages/Projects.tsx` — project list (from task 074)
- `web/src/pages/ProjectChat.tsx` — chat interface for a specific project

`ProjectChat` is like `Chat` but:
- Shows project name + app URL at the top
- Shows a "View App" button (opens `/apps/{workspaceId}` in a new tab)
- Shows session history in the sidebar (last 10 sessions with commit SHAs + status)
- Sends messages to `POST /api/projects/:id/sessions` instead of `/api/chat`
- Shows a "building..." state while the session container is running
- Locks the input while a session is active (one session at a time)
