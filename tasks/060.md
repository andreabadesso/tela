# 060 — Streaming Tool Execution

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 036, 042
**Estimated effort**: large (8-20h)

## Context

Currently the agent completes its full run before responding to the user — no per-token streaming from the Claude API. The cc-tips architecture docs describe a generator-based streaming loop where tools execute *during* model streaming, not after. Read-only tools run in parallel; write tools run exclusively. This dramatically reduces user-perceived latency.

**Source:** cc-tips `STREAMING_STATE_MACHINE.md`, `TOOL_PIPELINE.md`

## Deliverables

### 1. Generator-based query loop

Replace the current one-shot `agentService.process()` → full response pattern with a streaming loop:

```
while (true) {
  stream tokens from Claude API
  detect tool_use blocks mid-stream
  execute tools as soon as their input JSON is complete
  yield partial results to WebSocket
  feed tool results back into next turn
  check for terminal conditions (done, abort, error, context overflow)
}
```

### 2. StreamingToolExecutor

New class that manages concurrent tool execution during streaming:

- **Read-only tools** (search_vault, search_knowledge, list_notes, read_note): run in parallel
- **Write tools** (write_note, edit_note, Bash-equivalent): run exclusively (no other tools while executing)
- Results yielded in registration order for deterministic output
- Error isolation: one tool failure doesn't cancel siblings (unless it's a write tool)

### 3. WebSocket streaming protocol update

Current WS messages: `{ type: "thinking" | "text" | "tool_calls" | "done" | "error" }`

Add granular streaming types:
- `{ type: "token", data: "partial text" }` — per-token text streaming
- `{ type: "tool_start", data: { id, name, input } }` — tool execution began
- `{ type: "tool_progress", data: { id, status } }` — tool running indicator
- `{ type: "tool_result", data: { id, name, result } }` — tool completed
- Keep existing `done` and `error` types

### 4. Frontend streaming adapter

Update `useTelaRuntime()` to handle granular streaming:
- Show text tokens as they arrive (typewriter effect)
- Show tool execution indicators (spinner per tool)
- Show tool results inline as they complete
- Handle out-of-order tool completion gracefully

### 5. Concurrency rules

| Tool Type | Concurrency | Cancel Siblings on Error |
|-----------|------------|------------------------|
| search_* | parallel | no |
| read_* | parallel | no |
| list_* | parallel | no |
| write_* | exclusive | yes |
| edit_* | exclusive | yes |

### 6. Loop terminal states

Define clean exit conditions:
- `completed` — model says stop, no pending tools
- `aborted` — user cancelled mid-stream
- `error` — unrecoverable tool/API error
- `context_overflow` — trigger compaction (see task 061)
- `max_turns` — agent hit configured turn limit

## Architecture Decision

**Why not use Claude Agent SDK's built-in streaming?**

The SDK's `query()` function handles the turn loop internally. We need to either:
1. Use the SDK's streaming events (if exposed) and layer our executor on top
2. Drop to the raw Anthropic SDK for the query loop and keep the Agent SDK for tool definitions

Investigate which approach gives us control over mid-stream tool execution. The SDK may already emit events we can hook into — check `@anthropic-ai/claude-agent-sdk` streaming API first.

## Acceptance Criteria

- [ ] Text tokens stream to WebSocket as they're generated
- [ ] Tool execution starts mid-stream (not after full response)
- [ ] Read-only tools run in parallel
- [ ] Write tools run exclusively
- [ ] Frontend shows live token streaming + tool indicators
- [ ] Loop exits cleanly on all terminal conditions
- [ ] No regression: existing chat functionality works as before
- [ ] Measurable latency improvement: time-to-first-token < 500ms
