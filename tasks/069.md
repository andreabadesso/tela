# 069 — Startup Optimization & Parallel Prefetch

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: none
**Estimated effort**: small (1-4h)

## Context

All services initialize sequentially on startup. The cc-tips docs describe parallel prefetch patterns and deferred module loading that significantly reduce cold start time. This matters especially for scheduled jobs that spin up fresh instances.

**Source:** cc-tips `development_practices/STARTUP.md`

## Deliverables

### 1. Parallel service initialization

Current startup is sequential:
```
DB init → migrations → knowledge adapters → job registry → API server → WS server
```

Change to parallel where possible:
```
DB init → migrations → [
  knowledge adapters (async)  ─┐
  job registry (async)         ├── all in parallel
  API preconnect (async)       │
  ChromaDB health check (async)┘
] → API server → WS server
```

### 2. API preconnect

Fire-and-forget HEAD request to the Anthropic API during startup to warm the connection:

```typescript
// Don't await — just let it run in background
fetch('https://api.anthropic.com/', { method: 'HEAD' }).catch(() => {});
```

This overlaps TCP+TLS handshake with app initialization, shaving ~200-500ms off first request.

### 3. Deferred heavy module loading

Don't load these until first use:
- ChromaDB client (only needed when vector search is actually called)
- gray-matter (only when parsing Obsidian frontmatter)
- Any large MCP server SDKs

```typescript
// Lazy singleton pattern
let chromaClient: ChromaClient | null = null;
async function getChromaClient() {
  if (!chromaClient) {
    const { ChromaClient } = await import('chromadb');
    chromaClient = new ChromaClient({ path: process.env.CHROMA_URL });
  }
  return chromaClient;
}
```

### 4. Fast path for health checks

`GET /api/health` should respond without waiting for full initialization:
- If DB is ready, return basic health
- Don't block on knowledge adapters or external MCP connections

### 5. Startup timing metrics

Log timing for each initialization phase:

```
[startup] DB init: 45ms
[startup] Migrations: 120ms
[startup] Knowledge adapters: 230ms (parallel)
[startup] Job registry: 15ms (parallel)
[startup] API preconnect: fired (async)
[startup] Total: 280ms
```

## Acceptance Criteria

- [ ] Service initialization parallelized where possible
- [ ] API preconnect fires during startup
- [ ] ChromaDB and heavy modules loaded lazily on first use
- [ ] Health endpoint responds before full init
- [ ] Startup timing logged
- [ ] Cold start time reduced by >30%
- [ ] No behavior change: all services still work correctly
