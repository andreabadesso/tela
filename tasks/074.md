## Context

Right now the top-level entity for agent-built apps is a **workspace** — a container + a bind-mount directory. This conflates two distinct things:

- The **project** (what is being built, its identity, its git history)
- The **deploy context** (where the live app is served from, the bind mount on disk)

We need a `projects` entity that owns both, so non-technical users can think in terms of "my HR onboarding tool" or "our expense tracker" — not in terms of containers or workspaces.

A project has exactly one workspace (its deploy context — the bind mount that the app proxy serves from). A project has many sessions (each session is one agent run that clones the repo, does work, and pushes back).

## Data Model

### Migration: `021_projects.sql`

```sql
CREATE TABLE projects (
  id          TEXT PRIMARY KEY,
  name        TEXT NOT NULL,
  description TEXT,
  stack       TEXT NOT NULL DEFAULT 'insforge',   -- 'insforge' only for now

  -- Ownership & access
  owner_id    TEXT NOT NULL REFERENCES users(id),
  team_id     TEXT REFERENCES teams(id),
  visibility  TEXT NOT NULL DEFAULT 'private',    -- 'private' | 'team' | 'public'

  -- Git
  git_repo_slug  TEXT NOT NULL UNIQUE,            -- slug used in git URL: {slug}.git

  -- Deploy context (set after first successful session)
  workspace_id   TEXT REFERENCES workspaces(id),

  -- InsForge integration
  insforge_project_id TEXT,                       -- optional, linked InsForge project

  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_projects_owner    ON projects(owner_id);
CREATE INDEX idx_projects_team     ON projects(team_id);
CREATE INDEX idx_projects_visibility ON projects(visibility);
CREATE INDEX idx_projects_git_slug ON projects(git_repo_slug);
```

### Relationship to workspaces

The `workspaces` table is already used for the deploy context (bind mount + app proxy URL). A project has `workspace_id` pointing to one workspace. That workspace:
- Is created when the project is created
- Persists forever (survives session restarts, container deaths)
- Holds the `static_app_path` pointing to the last promoted build
- Is the entity the app proxy serves from (`/apps/{workspaceId}/*`)

Workspaces no longer need `agent_id` as a direct concept — they belong to a project.

### Project slug generation

`git_repo_slug` is derived from the project name: lowercase, spaces → hyphens, non-alphanumeric stripped.

```typescript
function toSlug(name: string, id: string): string {
  const base = name.toLowerCase().replace(/\s+/g, '-').replace(/[^a-z0-9-]/g, '');
  return `${base}-${id.slice(-6)}`;  // e.g., "expense-tracker-a3f2c1"
}
```

The short suffix ensures uniqueness without user-visible collisions.

## Project Lifecycle

### Create

```
POST /api/projects  { name, description?, team_id?, insforge_project_id? }

1. Generate project id (nanoid) + git_repo_slug
2. Create workspace record (bind mount dir on disk, status: 'created')
3. GitService.initRepo(slug)  → pushes initial commit
4. Insert project row with workspace_id
5. Return project with app_url: /apps/{workspaceId}
```

The workspace starts with `status: 'created'` (no container yet). The app proxy returns 503 until the first session deploys something.

### Update

```
PATCH /api/projects/:id  { name?, description?, visibility?, team_id?, insforge_project_id? }
```

Only owner or admin can update.

### Delete

```
DELETE /api/projects/:id

1. Destroy workspace (stops container if running, deletes bind mount)
2. GitService.deleteRepo(slug)
3. Delete project row (cascades to sessions)
```

Irreversible. Requires owner or admin.

## API Routes (`src/api/routes/projects.ts`)

### `GET /api/projects`

Returns projects accessible to the current user (own + team + public). Each item includes:
```typescript
{
  id, name, description, stack, visibility,
  git_repo_slug,
  app_url: `/apps/${workspace_id}`,
  session_count: number,
  last_session_at: string | null,
  last_commit_sha: string | null,
  workspace_status: 'created' | 'running' | 'paused' | 'destroyed',
  created_at, updated_at
}
```

### `POST /api/projects`

Creates project + workspace + git repo atomically (wrapped in DB transaction; git repo creation runs after transaction commits).

### `GET /api/projects/:id`

Full project detail including recent sessions (last 10).

### `PATCH /api/projects/:id`

Update name, description, visibility, team_id, insforge_project_id.

### `DELETE /api/projects/:id`

Destroy project. Requires confirmation: body must include `{ confirm: true }`.

## RBAC

Same model as workspaces:
- `private` → owner only
- `team` → any member of `team_id`
- `public` → any authenticated user
- Admins always have access

Access check is shared with workspace access check (they have the same visibility column values and the same logic applies).

- [ ] `POST /api/projects` creates workspace directory on disk + git repo + project row atomically
- [ ] `GET /api/projects` only returns projects the current user can access
- [ ] `GET /api/projects/:id` returns 404 for projects the user cannot access
- [ ] `DELETE /api/projects/:id` fails without `{ confirm: true }` body
- [ ] `DELETE /api/projects/:id` removes workspace directory from disk + git repo
- [ ] Visibility change to `team` requires `team_id` to be set
- [ ] Non-owner, non-admin cannot update or delete a project
- [ ] `app_url` in response resolves to the app proxy URL
- [ ] Projects page renders in the web UI with list of projects
