# 063 — Tool Execution Pipeline & Permission Hardening

**Status**: pending
**Phase**: 7 — Agent Runtime
**Dependencies**: 050
**Estimated effort**: medium (2-8h)

## Context

The MCP governance gateway exists but doesn't fully enforce at the per-tool-call level during agent execution. The cc-tips docs describe a 9-step execution pipeline per tool with defense-in-depth permission checks, pre/post hooks, and bypass-immune safety checks.

**Source:** cc-tips `TOOL_PIPELINE.md`, `HOOKS.md`

## Deliverables

### 1. Tool execution pipeline

Wrap every tool invocation (vault tools, knowledge tools, external MCP tools) in a standardized pipeline:

```
Step 1: Input validation (Zod schema)
Step 2: Permission check
  a. Deny rules (from mcp_policies denied_tools)
  b. Tool's own safety check (e.g., write tools require write access level)
  c. Bypass-immune checks (can NEVER be overridden):
     - Cannot delete knowledge source config
     - Cannot modify auth/RBAC tables via tools
     - Cannot exfiltrate credentials
  d. RBAC policy check (user role → mcp_policies → access_level)
  e. Rate limit check
Step 3: Pre-execution hooks (audit log entry: "tool X invoked by user Y")
Step 4: Execute tool
Step 5: Post-execution hooks (audit log entry: "tool X completed, result size N")
Step 6: Output validation (size limits, content classification check)
Step 7: Return result
```

### 2. Bypass-immune safety checks

Define a set of operations that NO permission level can override:

- Deleting database tables or schema
- Reading/writing encryption keys
- Modifying auth configuration
- Accessing other users' credentials
- Writing to system paths outside the configured scope

These checks are hardcoded, not policy-driven — they can't be misconfigured.

### 3. Per-invocation classification

Same tool, different inputs, different treatment:

```typescript
// read_note("meeting-notes.md") → routine, auto-allow
// write_note("config.json") → elevated, requires write access level
// search_vault("salary") → sensitive query, log with classification
```

Classify based on:
- Tool type (read vs write vs search)
- Input patterns (sensitive keywords in search queries)
- Target resource (system files vs user files)

### 4. Tool execution hooks for observability

Pre/post hooks that fire for every tool call:
- Pre: log to audit_logs, check rate limits, validate input
- Post: log result size, duration, success/failure
- Error: log error details, check circuit breaker

### 5. Output size limits

Large tool results (>25K tokens) should be:
1. Persisted to a temp file
2. Replaced in conversation with a truncated preview (first 2KB)
3. Model told: "Full result saved to {path}, showing first 2KB"

This prevents tool results from dominating the context window.

## Acceptance Criteria

- [ ] Every tool call goes through the standardized pipeline
- [ ] Bypass-immune checks cannot be overridden by any policy
- [ ] Per-invocation classification logs sensitive queries
- [ ] Pre/post hooks fire for every tool call
- [ ] Large tool results truncated with disk persistence
- [ ] Audit trail complete: every tool call logged with user, input, result size, duration
- [ ] No regression: existing tool functionality works as before
